# Write Script — 발표 대본 작성/개선

리서치 결과를 바탕으로 청중을 사로잡는 발표 대본을 작성하거나, 기존 대본을 분석하고 개선한다. 두 가지 모드를 지원한다:
- **신규 작성 모드**: `_workspace/00_original_script.md`가 없을 때. 아래 "대본 구조 설계" 섹션을 따른다.
- **개선 모드**: `_workspace/00_original_script.md`가 있을 때. 아래 "개선 모드" 섹션을 따른다.

## 대본 구조 설계

### 전체 흐름: 3막 구조

| 구간 | 비율 | 목적 |
|------|------|------|
| **오프닝** | 10-15% | 관심 잡기, 문제 제기 |
| **본론** | 70-80% | 핵심 메시지 전달 |
| **클로징** | 10-15% | 정리, 행동 촉구 |

### 오프닝 설계 (가장 중요)

첫 30초에 청중의 관심을 잡아야 한다. 다음 훅 중 하나를 선택한다:

- **충격적 통계**: "여러분이 이 발표를 듣는 30분 동안, X가 Y만큼 발생합니다"
- **질문**: "여러분은 마지막으로 Z한 적이 언제인가요?"
- **스토리**: 2-3문장의 짧은 일화
- **역설**: "우리가 X라고 믿는 것은 사실 Y입니다"

오프닝에서 피할 것: 자기소개로 시작, "오늘 발표할 내용은...", 목차 나열

### 슬라이드별 대본 작성 원칙

- **1슬라이드 = 1메시지**: 하나의 슬라이드에서 하나의 핵심만 전달한다
- **구어체**: 문어체를 피하고 실제 말하는 것처럼 작성한다
- **전환 멘트**: 슬라이드 사이에 자연스러운 연결을 만든다. "이것이 왜 중요할까요? 바로 다음 내용과 연결됩니다"
- **핵심 키워드**: 전체 문장 암기용이 아니라 핵심 키워드와 흐름 중심으로 작성한다
- **시간 표시**: 각 슬라이드에 예상 소요 시간을 명시한다

### 참여 포인트 설계

5-7분마다 청중 참여 요소를 하나씩 배치한다:
- 질문 던지기 (수사적 또는 실제)
- 손들기 요청
- 잠깐의 생각 시간
- 실시간 투표/퀴즈 제안
- 주변 사람과 짧은 대화

### 클로징 설계

- 핵심 메시지를 3가지 이하로 요약
- 청중이 내일 당장 할 수 있는 구체적 행동 제안 (Call to Action)
- 오프닝의 훅을 회수하면 완성도가 올라간다

## 출력 형식

```markdown
# 발표 대본: [제목]

## 발표 정보
- 총 시간: N분
- 슬라이드 수: N장
- 대상 청중: [설명]

---

## 슬라이드 1: [제목]
**핵심 메시지:** [한 줄]
**시간:** N분

**대본:**
[발표 멘트 — 구어체로]

**전환:** [다음 슬라이드로 넘어가는 멘트]

---

## 슬라이드 2: [제목]
...
```

## 개선 모드

`_workspace/00_original_script.md`에 기존 대본이 있으면 이 모드로 동작한다. 핵심 원칙: **원작자의 의도와 스타일을 존중하면서 객관적으로 약한 부분만 개선한다.**

### 1단계: 기존 대본 분석

기존 대본을 읽고 다음을 파악한다:

| 분석 항목 | 파악할 것 |
|----------|----------|
| **전체 구조** | 몇 막 구성인가, 논리 흐름은 어떤가, 시간 배분은 적절한가 |
| **관통 내러티브** | 전체를 관통하는 핵심 메시지/메타포가 있는가 (있다면 반드시 보존) |
| **강점** | 잘 작동하는 부분 (훅, 전환, 사례, 어조). 명시적으로 기록한다 — 실수로 제거 방지 |
| **약점** | 논리 비약, 근거 부족, 전환 부자연, 참여 포인트 부족, 정보 과밀 구간 |
| **어조/스타일** | 원작자의 말투, 격식 수준, 유머 사용 패턴 |
| **인용 데이터** | 대본에 언급된 수치, 출처, 사례 목록 (researcher 검증 결과와 대조) |

분석 결과를 `_workspace/02_scriptwriter_analysis.md`에 저장한다.

### 2단계: 개선 계획 수립

분석을 바탕으로 개선 항목을 우선순위별로 정리한다:

- **Critical**: 논리 비약, 잘못된 데이터, 핵심 메시지 불명확
- **Important**: 전환 부자연, 참여 포인트 부족, 시간 배분 불균형
- **Nice-to-have**: 표현 다듬기, 비유 추가, 사례 보강

### 3단계: 대본 개선 실행

개선 원칙:
- **강점 보존**: 분석에서 "잘 작동하는 부분"으로 기록한 것은 변경하지 않는다
- **관통 내러티브 유지**: 전체를 관통하는 메시지나 메타포가 있으면 그 구조를 유지하고 강화한다
- **어조 일관성**: 원작자의 말투를 모방한다. 갑자기 격식이 바뀌거나 다른 사람이 쓴 것처럼 보이면 안 된다
- **최소 개입**: 불필요한 재작성을 하지 않는다. 문제가 없는 문장은 그대로 둔다
- **researcher 검증 반영**: 잘못된 데이터는 수정하고, 더 강력한 데이터가 있으면 교체하되 원본의 주장 방향은 유지
- **변경 추적**: 주요 변경 사항을 대본 상단에 "## 변경 이력" 섹션으로 남긴다

### 개선 모드 출력 형식

```markdown
# 발표 대본: [제목] (개선본)

## 변경 이력
- [Critical] 슬라이드 N: [무엇을 왜 변경했는지]
- [Important] 슬라이드 M: [무엇을 왜 변경했는지]
- ...

## 발표 정보
- 총 시간: N분
- 슬라이드 수: N장
- 대상 청중: [설명]
- 원본 대비 변경률: [대략적 비율]

---

## 슬라이드 1: [제목]
**핵심 메시지:** [한 줄]
**시간:** N분
**변경:** [없음 | 수정 내용 한 줄 요약]

**대본:**
[발표 멘트]

**전환:** [다음 슬라이드로 넘어가는 멘트]
```

## 피드백 반영 시 주의사항

audience-reviewer의 피드백을 반영할 때:
- Critical 피드백은 반드시 반영
- Important 피드백은 대본의 전체 흐름을 해치지 않는 범위에서 반영
- "잘된 점"으로 명시된 부분은 수정하지 않도록 주의
- 수정 사항을 slide-builder에게 SendMessage로 알린다
